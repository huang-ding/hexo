服务容错机制-hystrix

微服务中服务间相互依赖，一个请求可能是多个微服务协调处理。可能出现响应慢，超时，网络不同，或者说服务内部程序问题，当出现这些情况时，需要用到这个服务的所有微服务会阻塞在请求这个位置，请求一旦被大量阻塞会占用大量资源，就可能导致整个系统雪崩。

构建弹性应用防止出现系统雪崩的思路

  • 服务限流

    服务消费者限制自身对某一服务能够发起的并发请求数量，超过数量则不调用；如1毫秒只接受100个请求，当超过则不调用

  • 服务熔断

    熔断状态下，服务消费者不会发起对某一服务的调用；返回错误或返回服务降级后的结果

  • 服务降级

    请求异常下，程序制定执行指定的降级策略，如代码中的try catch代码块

spring-cloud提供了hystrix对服务进行保护，从而进行服务降级等操作

Hystrix的初始化流程

  1. HystrixCommand构造函数—>AbstractHystrixCommand

    1. commandGoup：命令分组名称（服务名）
    2. commandKey：命令名称（具体接口、业务）
    3. properties：加载配置
    4. metrics：度量器，收集数据
    5. circuitBreaker：熔断器（标识命令是否已熔断）
    6. threadPool：线程池，资源隔离的手段。与分组对应
Hystrix的执行流程

  1. 通过HystrixCommand、HystrixObservableCommand发起请求

  2. available in cache 判断缓存，有则直接返回结果

  3. circuit breaker open？ 断路器是否打开，打开则执行fallback，进行服务降级

  4. Semaphore/Thread pool rejected？ 判断信号量或线程池是否可以通行，满了进行fallback，同时进行metrics进行统计，看是否满足打开circuitBreaker的条件，满足则打开熔断器

  5. construct or run() 执行run()方法，执行的业务代码

  6. execution fails? 执行是否成功，或是否超时，否进行fallback，同时进行metrics进行统计，看是否满足打开circuitBreaker的条件，满足则打开熔断器（此阶段相当于java的 Future 和 Callable，源码中使用了rxjava响应式编程）
